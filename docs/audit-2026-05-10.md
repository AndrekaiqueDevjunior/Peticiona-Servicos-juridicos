# Auditoria & Correção 2026-05-10 — Planos, Avulsos, Compras, Uploads, Admin

Documento de entrega da rodada de correção de produção. Cobre os 7 pontos
solicitados (planos, serviços avulsos, compra, upload, admin clientes,
remoção de mocks/hardcode e regras de produção).

---

## 1. Resumo dos bugs encontrados

### Planos
- Frontend hardcodava `PLAN_PRICE_PER_SERVICE`, `BASE_FEATURES` e
  `PLAN_EXTRA_FEATURES` em `Pricing.tsx`. O backend já tinha as colunas
  `is_highlighted`, `cta_label`, `price_per_service_cents` e `features_json`,
  mas os campos não eram consumidos pelo frontend.
- Badge "Mais escolhido" estava amarrado a `index === 1`, ignorando o
  campo `is_highlighted` do banco.
- Texto do botão era `Adquirir ${plan.name}` — não respeitava o
  `cta_label` configurado.
- Tipos do frontend (`PublicPlan`, `AdminPlan`) não expunham os novos
  campos.

### Serviços avulsos
- Modelo SQLAlchemy `ServiceCatalogItem` não tinha a coluna
  `delivery_label`, embora a migração de produção já a tivesse adicionado
  ao banco — qualquer SELECT carregava o registro sem o campo.
- Frontend determinava se o serviço era "Express" via regex
  `/express/i` no código/título, em vez de usar o `delivery_label` real
  vindo do banco.
- Admin não conseguia editar `delivery_label` em `AdminPlans`.

### Fluxo de compra
- Backend já é fonte da verdade: `_catalog_entry()` em
  `checkout_service.py` busca preço pelo `code` no banco e ignora o
  preço enviado pelo frontend.
- Webhook idempotente via `PaymentEvent.gateway_event_id` — ok.
- Crédito é liberado por `_release_order()` apenas em status `paid` e
  com guarda de `released_at`, evitando duplicação.
- Sem alterações funcionais; apenas auditado.

### Upload de documentos nos pedidos
- `DEFAULT_UPLOAD_FOLDER` em `core/config.py` apontava para
  `backend/backend/app/uploads` (caminho duplicado errado). Em produção
  o env `UPLOAD_FOLDER` mascarava o bug, mas dev/staging sem o env
  acabavam criando uploads num diretório fora do esperado.
- Frontend `NewRequestDialog.tsx` aceitava `image/*` (qualquer formato
  de imagem) e usava limite de **200 MB**, enquanto o backend permite
  apenas PDF/DOC/DOCX/PNG/JPG/JPEG e usa `MAX_UPLOAD_MB=20` em
  produção.

### Admin – clientes
- Tipo `AdminClient` no frontend não expunha CPF, OAB separada da UF
  nem versões formatadas.
- Tabela `AdminClients.tsx` não exibia CPF nem UF da OAB.
- `EditClientDialog.tsx` não permitia editar CPF.
- Backend serializer `_serialize_client` já retorna todos os campos
  necessários (`cpf`, `cpf_formatado`, `oab_number`, `oab_uf`,
  `telefone_formatado`, `cadastrado_em`), mas faltava `cadastrado_em_iso`
  e o atalho `full_name`/`phone` para o dialog.
- O update do admin (`_create_or_update_user`) não tratava entrada
  separada de `oab` + `oab_uf`, exigindo o campo combinado
  `oab_number`.

### Mocks/hardcode residuais
- Constantes `BASE_FEATURES`, `PLAN_EXTRA_FEATURES`,
  `PLAN_PRICE_PER_SERVICE` removidas do `Pricing.tsx`.
- `BuyCreditsDialog.tsx` reescrito para usar features, CTA, destaque e
  delivery_label vindos do backend.
- `lib/pricing.ts` mantém um `DEFAULT_PRICING` exclusivamente como
  *fallback* para o caso da API estar indisponível (não é exibido em
  tela enquanto o catálogo carrega; foi mantido para evitar tela em
  branco em caso de falha de rede).

## 3. Endpoints corrigidos / consolidados

| Endpoint | Comportamento |
|---|---|
| `GET /api/plans` | Retorna planos ativos com `features`, `is_highlighted`, `cta_label`, `price_per_service_cents`, ordenados pelo preço crescente. |
| `GET /api/catalog` | Catálogo público de avulsos com `delivery_label` e `description`, apenas itens ativos. |
| `GET /api/admin/plans` | Lista planos + `single_services`, todos com `is_highlighted`, `cta_label`, `delivery_label`. |
| `PUT /api/admin/plans/{id}` | Aceita `is_highlighted`, `cta_label`, `features` (JSON), `price_per_service_cents`. |
| `PUT /api/admin/services/{id}` | Aceita `delivery_label`. |
| `GET /api/admin/clients` | Inclui `cpf`, `cpf_formatado`, `oab_number`, `oab_uf`, `telefone_formatado`, `cadastrado_em_iso`. |
| `PATCH /api/admin/clients/{id}` | Aceita `cpf`, `oab`, `oab_uf` (separados) ou `oab_number` (combinado). |
| `POST /api/checkout/create-order` | Lê preço e nome do banco via `service_id`. |
| `POST /api/checkout/create-payment` | Recusa dados brutos de cartão (PCI). Gera idempotency key por tentativa. |
| `POST /api/orders/{id}/documents` | Upload multipart/form-data, campo `documents`. Aceita PDF/DOC/DOCX/PNG/JPG/JPEG/TXT até `MAX_UPLOAD_MB`. |

Sem alteração de assinatura/quebra de contrato — apenas adição de
campos na resposta.

## 5. Como testar (staging/produção)

### Pré-requisitos
1. Aplicar as migrations:
   ```bash
   psql "$DATABASE_URL" -f migrations/2026-05-10-fixes.sql
   psql "$DATABASE_URL" -f migrations/2026-05-10-seed-canonical.sql
   ```
2. Rebuild do backend (model agora declara `delivery_label`).
3. Rebuild do frontend.

### Plano de teste manual

#### Planos
- [ ] Abra a landing page sem login. Confirmar que aparecem
      Essencial (R$480), Profissional (R$750, com badge "Mais
      escolhido"), Estratégico (R$2.800), nessa ordem.
- [ ] Botões devem dizer "Adquirir Essencial / Profissional /
      Estratégico".
- [ ] Cada card mostra os 5 ou 6 benefícios oficiais e o "R$ X por
      serviço" correspondente.
- [ ] No admin → /admin/planos, alterar `monthly_price_cents` do
      Essencial para 49000, salvar, recarregar a landing → o preço
      muda para R$490.
- [ ] Reverter para 48000.

#### Serviços avulsos
- [ ] Na landing (seção "Avulsos") devem aparecer só Petição R$180,
      Recurso R$200, Petição Express R$220 (badge 24h), Recurso
      Express R$250 (badge 24h).
- [ ] Os 12 serviços legados não devem aparecer.
- [ ] Em /admin/planos → "Servicos avulsos", editar `delivery_label`
      do "Recurso Express" para "Entrega expressa em 24h", salvar e
      recarregar a landing → texto reflete imediatamente.

#### Compra de créditos
- [ ] Login como cliente.
- [ ] /area-cliente/saldos → "Comprar créditos" → escolher Plano
      Profissional → checkout em PIX.
- [ ] Pagar e confirmar que ao retornar para `/area-cliente/saldos` o
      saldo mostra R$ 750,00 disponíveis.
- [ ] Repetir tentando manipular o `service_id` no devtools (ex:
      mudar `code=profissional` para um valor inválido) → API responde
      404.
- [ ] Reenviar o webhook duplicado pela API da Pagar.me — confirmar
      que o crédito não é creditado duas vezes (`PaymentEvent` com o
      mesmo `gateway_event_id` retorna `duplicate=true`).

#### Upload de documentos
- [ ] Login como cliente → "Novo pedido" → anexar um PDF, um DOCX,
      um JPG; criar pedido. Tudo deve subir.
- [ ] Tentar anexar um `.exe` ou `.zip` → frontend rejeita com toast
      "Formato não suportado".
- [ ] Tentar anexar arquivo > 50 MB → toast "Arquivo muito grande".
- [ ] Login como admin → /admin/pedidos → abrir o pedido recém-criado
      → baixar cada anexo (`/api/documents/{id}/download`).
- [ ] Logout/login do cliente → o arquivo continua aparecendo no
      pedido.

#### Admin clientes
- [ ] Cadastrar um novo cliente pelo /signup com nome, telefone, CPF,
      OAB, UF.
- [ ] Em /admin/clientes confirmar que todas as colunas aparecem
      preenchidas (CPF formatado, número OAB, UF, telefone

- [ ] Editar o cliente alterando o telefone e a UF da OAB. Salvar e
      reabrir.
- [ ] Verificar via `curl` que `GET /api/admin/clients/{id}` **não**

## 6. Riscos remanescentes

1. **Catálogo de petições por área (`_resolve_service_item`)** — o
   matcher por nome continua usando heurística textual. Para o fluxo
   "Petição inicial comum" (grupo B no front, mas título com a palavra
   "Petição"), o resolver pode acabar caindo no fallback. Não é parte
   desta auditoria, mas vale uma rodada futura.
2. **Soft-delete de cliente** (`delete_admin_client`) só seta
   `is_active=false`. Permanece intencional — manter histórico de
   pedidos.
3. **Snapshot pyc-only** — vários services (`auth_service`,
   `audit_service`, `dashboard_service`, `staff_service`,
   `credit_payment_service`, `split_payment_service`) só estão como
   `.pyc`. Não foram alterados. As correções acima respeitam contratos
   públicos.
4. **`lib/pricing.ts` ainda mantém `DEFAULT_PRICING`** — apenas
   fallback off-line. Caso a API caia, o cliente vê valores default;
   se isso for inaceitável, podemos retirar e mostrar erro.
